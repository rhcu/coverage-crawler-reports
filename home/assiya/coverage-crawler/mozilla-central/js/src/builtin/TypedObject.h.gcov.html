      18             :  * -------------
      19             :  * Typed Objects
      20             :  * -------------
      21             :  *
      22             :  * Typed objects are a special kind of JS object where the data is
      23             :  * given well-structured form. To use a typed object, users first
      24             :  * create *type objects* (no relation to the type objects used in TI)
      25             :  * that define the type layout. For example, a statement like:
      26             :  *
      27             :  *    var PointType = new StructType({x: uint8, y: uint8});
      28             :  *
      29             :  * would create a type object PointType that is a struct with
      30             :  * two fields, each of uint8 type.
      31             :  *
      32             :  * This comment typically assumes familiary with the API.  For more
      33             :  * info on the API itself, see the Harmony wiki page at
      34             :  * http://wiki.ecmascript.org/doku.php?id=harmony:typed_objects or the
      35             :  * ES6 spec (not finalized at the time of this writing).
      36             :  *
      37             :  * - Initialization:
      38             :  *
      39             :  * Currently, all "globals" related to typed objects are packaged
      40             :  * within a single "module" object `TypedObject`. This module has its
      41             :  * own js::Class and when that class is initialized, we also create
      42             :  * and define all other values (in `js::InitTypedObjectModuleClass()`).
      43             :  *
      44             :  * - Type objects, meta type objects, and type representations:
      45             :  *
      46             :  * There are a number of pre-defined type objects, one for each
      47             :  * scalar type (`uint8` etc). Each of these has its own class_,
      48             :  * defined in `DefineNumericClass()`.
      49             :  *
      50             :  * There are also meta type objects (`ArrayType`, `StructType`).
      51             :  * These constructors are not themselves type objects but rather the
      52             :  * means for the *user* to construct new typed objects.
      53             :  *
      54             :  * Each type object is associated with a *type representation* (see
      55             :  * TypeRepresentation.h). Type representations are canonical versions
      56             :  * of type objects. We attach them to TI type objects and (eventually)
      57             :  * use them for shape guards etc. They are purely internal to the
      58             :  * engine and are not exposed to end users (though self-hosted code
      59             :  * sometimes accesses them).
      60             :  *
      61             :  * - Typed objects:
      62             :  *
      63             :  * A typed object is an instance of a *type object* (note the past participle).
      64             :  * Typed objects can be either transparent or opaque, depending on whether
      65             :  * their underlying buffer can be accessed. Transparent and opaque typed
      66             :  * objects have different classes, and can have different physical layouts.
      67             :  * The following layouts are possible:
      68             :  *
      69             :  * InlineTypedObject: Typed objects whose data immediately follows the object's
      70             :  *   header are inline typed objects. The buffer for these objects is created
      71             :  *   lazily and stored via the compartment's LazyArrayBufferTable, and points
      72             :  *   back into the object's internal data.
      73             :  *
      74             :  * OutlineTypedObject: Typed objects whose data is owned by another object,
      75             :  *   which can be either an array buffer or an inline typed object. Outline
      76             :  *   typed objects may be attached or unattached. An unattached typed object
      77             :  *   has no data associated with it. When first created, objects are always
      78             :  *   attached, but they can become unattached if their buffer becomes detached.
      79             :  *
      80             :  * Note that whether a typed object is opaque is not directly
      81             :  * connected to its type. That is, opaque types are *always*
      82             :  * represented by opaque typed objects, but you may have opaque typed
      83             :  * objects for transparent types too. This can occur for two reasons:
      84             :  * (1) a transparent type may be embedded within an opaque type or (2)
      85             :  * users can choose to convert transparent typed objects into opaque
      86             :  * ones to avoid giving access to the buffer itself.
      87             :  *
      88             :  * Typed objects (no matter their class) are non-native objects that
